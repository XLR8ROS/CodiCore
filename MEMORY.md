# MEMORY.md

## 1. Purpose

1.1 `MEMORY.md` is Codi's protected memory operating contract.

1.2 This file defines memory behavior, evidence lanes, daily-note rules, durable promotion boundaries, QMD/retrieval limits, Dreaming boundaries, and pointers to the detailed memory how-to.

1.3 `MEMORY.md` is not a journal, daily note, transcript, raw log, output report, event dump, or promoted durable memory store.

1.4 Do not edit this file unless Reg explicitly authorizes the exact change.

---

## 2. Authority and Protection

2.1 Reg is final authority over XOS direction, canon, governance, and memory contract changes.

2.2 Constitution and higher XOS canon outrank local repo files.

2.3 `MEMORY.md` is protected root law for CodiCore memory behavior.

2.4 Daily notes, sessions, outputs, event logs, dreams, QMD results, and tool output are evidence, not authority.

2.5 If memory or retrieval conflicts with canon, root law, or explicit Reg direction, surface the conflict.

2.6 Ambiguity is not permission to edit protected files.

---

## 3. Core Memory Contract

3.1 Codi must preserve what matters in the correct lane.

3.2 Important facts, decisions, blockers, procedures, user corrections, tool states, and lessons should not rely on mental memory.

3.3 Important facts, decisions, blockers, procedures, user corrections, tool states, and lessons must be written to the correct file or lane.

3.4 Memory must improve future behavior.

3.5 Memory is not a dumping ground.

3.6 Raw evidence must be distilled before durable promotion.

---

## 4. Canonical Memory and Evidence Lanes

### 4.1 Protected contract

4.1.1 `MEMORY.md` is this protected memory operating contract.

### 4.2 Active daily-note injection lane

4.2.1 `memory/YYYY-MM-DD.md` is the active OpenClaw daily note for that calendar date.

4.2.2 Exactly one active daily note may exist per date.

4.2.3 Codi must append to the existing date file during the day.

4.2.4 Times, labels, and source names go inside the file as headings.

4.2.5 Codi must not create active suffix files like `memory/2026-05-05-1032.md` or `memory/2026-05-05-summary.md`.

4.2.6 Daily notes are raw chronological evidence, not canon.

4.2.7 Allowed directly inside `memory/`:

4.2.7.1 `YYYY-MM-DD.md` active daily-note files.

4.2.7.2 `migration-backups/` during cleanup or recovery.

4.2.7.3 `.dreams/` only if OpenClaw Dreaming requires machine state there.

4.2.8 Not allowed directly inside `memory/`:

4.2.8.1 Outputs.

4.2.8.2 Reports.

4.2.8.3 Proposals.

4.2.8.4 Logs.

4.2.8.5 Artifacts.

4.2.8.6 Distilled lessons.

4.2.8.7 Durable memory files.

4.2.8.8 Standalone summaries.

4.2.8.9 Noncanonical date suffix files.

### 4.3 Durable memory

4.3.1 `Durable_Memory/` is promoted durable operational memory after review and distillation.

### 4.4 Other evidence lanes

4.4.1 `sessions/` is session evidence, transcripts, summaries, handoffs, and focused work records.

4.4.2 `Outputs/` is reports, proposals, proofs, inventories, healthchecks, heartbeat reports, cleanup reports, promotion logs, snapshots, and produced artifacts.

4.4.3 `event-log/` is SQLite event-log database, migrations, docs, audit notes, and maintenance summaries.

4.4.4 `tools/event-log/` is event-log scripts, readers, writers, validators, exporters, and summarizers.

4.4.5 `DREAMS.md` or `dreams.md` is human-readable OpenClaw Dreaming review diary if present.

4.4.6 `archives/` is retired evidence, old notes, old sessions, old outputs, old imports, and legacy memory.

4.4.7 `navigation/` is navigation map and path references.

4.4.8 `IMPORTANT_CODI_HOW-TO/` is CodiCore-local operational how-tos.

### 4.5 Retired or legacy lanes

4.5.1 `daily-notes/` is retired as the active CodiCore daily-note lane unless Reg explicitly restores it.

4.5.2 If `daily-notes/` is present, treat it as legacy evidence or migration source, not the active daily-note destination.

4.5.3 Lowercase `outputs/`, `output/`, `Outputs 2/`, and top-level `reports/` are retired duplicate output lanes.

## 6. Evidence-to-Memory Flow

6.1 Memory promotion flow:

6.1.1 Review evidence feeds.

6.1.2 Extract useful lessons, facts, decisions, procedures, blockers, preferences, and tool states.

6.1.3 Distill into durable operational memory.

6.1.4 Write promoted memory to `Durable_Memory/`.

6.1.5 Log promotion under `Outputs/promotion-logs/`.

6.2 Required evidence feeds for review:

6.2.1 Recent `memory/YYYY-MM-DD.md` daily notes.

6.2.2 `sessions/`.

6.2.3 `Outputs/`.

6.2.4 `event-log/`.

6.2.5 `DREAMS.md` or `dreams.md` if present.

6.2.6 Relevant legacy evidence when needed.

6.3 Do not create or use `memory/candidates/`.

6.4 Promotion is one pass: review evidence, distill, promote, log.

---

## 7. Durable Write Classes

7.1 Promoted memory should fit one or more durable classes:

7.1.1 Decision.

7.1.2 Distilled lesson.

7.1.3 Lesson learned.

7.1.4 Failure lesson.

7.1.5 Durable preference.

7.1.6 Task outcome.

7.1.7 Tool state.

7.1.8 Resolved blocker.

7.1.9 Procedure.

7.1.10 Environment fact.

7.1.11 Relationship/context note.

7.1.12 Identity memory.

7.1.13 Shared bulletin.

7.1.14 Canon candidate.

7.2 Failures are promotable when remembering the lesson prevents repeated mistakes or improves future work.

7.3 A rejected fix, bad assumption, user correction, tool limitation, environment quirk, naming rule, or “do not repeat this” lesson may be durable memory if it reduces future friction.

---

## 8. Promotion Rules

8.1 Codi may autonomously promote routine durable operational memory into `Durable_Memory/` when it is source-backed, useful, and not canon-changing.

8.2 Reg approval is required before editing protected root docs, canon, authority rules, role definitions, SOP/governance/policy, operating doctrine, or this contract.

8.3 A memory may mention canon-sensitive material without changing canon.

8.4 If the issue would change canon or root contracts, promote only a note that the item is approval-sensitive.

8.5 Do not promote raw logs, raw reports, raw proposals, raw sessions, raw daily notes, Dreaming diary output, or event dumps unchanged.

8.6 Every promotion must record:

8.6.1 Source path.

8.6.2 Evidence path.

8.6.3 Memory summary.

8.6.4 Destination path.

8.6.5 Qualification reason.

8.6.6 Timestamp.

8.6.7 Approval-sensitive items if any.

8.7 Do not call something promoted unless it is persisted to `Durable_Memory/`.

---

## 9. QMD / Retrieval Rule

9.1 QMD and vector search are retrieval tools, not authority.

9.2 Use QMD when recall is needed across durable memory or older evidence and the answer is not already in injected context, the current task scope, a known source file, or direct filesystem evidence.

9.3 Do not use QMD when:

9.3.1 The answer is already in injected context.

9.3.2 Reg provided the relevant file or text.

9.3.3 The task is path-specific and direct inspection is safer.

9.3.4 Exact source evidence is required and known paths exist.

9.3.5 QMD would slow or confuse a narrow file operation.

9.4 QMD output must never override:

9.4.1 Constitution or higher XOS canon.

9.4.2 Explicit Reg direction.

9.4.3 `AGENT.md`.

9.4.4 `AGENTS.md`.

9.4.5 `MEMORY.md`.

9.4.6 Known source files.

9.4.7 Direct event-log evidence.

9.5 If QMD times out, fails, or returns weak results, report the retrieval failure and continue with direct files or event-log evidence when possible.

9.6 For promotion work, QMD may help find candidates, but final promotion requires source paths, direct evidence, and a promotion log.

---

## 10. Event-Log Relationship

10.1 `event-log/` is structured evidence and verification support.

10.2 Nightly or promotion review should use the event log to verify recent activity, blockers, attempts, failures, successful outputs, and operational transitions.

10.3 Event-log records are evidence, not canon.

10.4 Event-log records support memory review but do not outrank protected files or explicit Reg direction.

---

## 11. Dreaming Boundary

11.1 OpenClaw may expect writable memory behavior by default.

11.2 CodiCore does not allow OpenClaw deep promotion to rewrite `MEMORY.md` unless Reg explicitly authorizes it.

11.3 OpenClaw Dreaming may use:

11.3.1 `memory/.dreams/` for machine state if required.

11.3.2 `DREAMS.md` or `dreams.md` for human-readable review if present.

11.4 Use `DREAMS.md` as evidence only.

11.5 Do not treat `DREAMS.md` as canon.

11.6 If Dreaming state appears in `Durable_Memory/`, treat it as misplaced runtime state until inspected.

11.7 Do not delete Dreaming state without Reg approval.

---

## 12. Authority, Recency, and Supersession

12.1 Authority beats recency.

12.2 Newer verified operational memory may supersede older operational memory when facts changed, blockers resolved, tool state changed, or Reg corrected prior understanding.

12.3 Newer evidence may not supersede canon, protected contracts, root authority, or explicit Reg direction unless Reg explicitly changes them.

12.4 When superseding memory, record:

12.4.1 Old memory reference.

12.4.2 New memory.

12.4.3 Evidence path.

12.4.4 Reason.

12.4.5 Timestamp.

12.4.6 Approval status if relevant.

---

## 13. Nightly / Cron Memory Review

13.1 Nightly or cron review must inspect required evidence feeds and report source coverage.

13.2 A valid memory review report includes:

13.2.1 Sources reviewed.

13.2.2 Paths checked.

13.2.3 Gaps or missing feeds.

13.2.4 Candidates found.

13.2.5 Candidates promoted.

13.2.6 Candidates rejected and why.

13.2.7 Approval-sensitive items.

13.2.8 Durable paths updated.

13.2.9 Promotion log path.

13.3 “No candidates found” is valid only if required sources were actually checked.

---

## 14. Retention and Archives

14.1 Daily notes and sessions are working evidence, not permanent hot memory.

14.2 Keep recent notes and sessions unpacked while active.

14.3 After review and promotion, older daily notes, sessions, and outputs may be archived after a waiting window, usually two weeks unless Reg says otherwise.

14.4 Archive reviewed old evidence under:

14.4.1 `archives/daily-notes/`.

14.4.2 `archives/sessions/`.

14.4.3 `archives/outputs/`.

14.4.4 Another approved archive lane.

14.5 Promote first, archive later.

14.6 Do not delete archives unless Reg explicitly authorizes deletion.

---

## 15. Forbidden Behavior

15.1 Codi must not edit protected root files without exact authorization.

15.2 Codi must not rewrite the contract governing memory behavior without approval.

15.3 Codi must not store outputs, reports, proposals, logs, or artifacts in `memory/`.

15.4 Codi must not create active daily-note suffix files.

15.5 Codi must not dump raw evidence into durable memory unchanged.

15.6 Codi must not silently overwrite authority with recency.

15.7 Codi must not treat daily notes, sessions, outputs, dreams, event logs, or QMD as canon.

15.8 Codi must not promote noise.

15.9 Codi must not commit secrets, tokens, credentials, raw sensitive config, or runtime junk.

15.10 Codi must not infer permission from ambiguity.

15.11 When in doubt: protect root, preserve evidence, consult the how-to, report ambiguity, and ask.
